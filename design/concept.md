# Concept — Crystal Vanguard

## 一文コンセプト

円形アリーナの中心に立つヒーローが、固定俯瞰カメラで四方から迫る群れを見渡しながら、自動攻撃（最寄り1体への単体ヒット・周期発動）で間引きつつ、紙一重のダッシュ回避だけで生き残り続ける3Dウェーブサバイバー。

## ピラー

| ID | 名前 | 意味 | 裁定に使う判断例 |
|---|---|---|---|
| P-01 | 紙一重回避 | ダッシュ回避の短い無敵窓によって「際どく躱せた」実感を作る。回避が安全すぎても脆すぎても壊れる。 | 被弾を減らす追加手段を検討するとき、無敵時間の延長や連続発動の許可ではなく、クールダウン短縮や移動距離調整で対応する。ダッシュを長時間無敵化・多段発動化する提案は却下する。 |
| P-02 | 照準ゼロの自動攻撃 | 攻撃入力を完全に排除し、プレイヤーの認知負荷を「避ける判断」だけに絞る。 | 新しい攻撃手段を検討するとき、手動照準・ターゲット選択・攻撃ボタンを要求する機能は却下する。自動攻撃の範囲/頻度/威力パラメータの強化のみ許可する。 |
| P-03 | 群れ密度の圧力 | 敵種の多様化ではなく、物量と速度がウェーブごとに上がることで「逃げ場がなくなっていく」緊張を作る。 | 難度を上げたいとき、新しい敵AI/敵種追加ではなく、まずスポーン間隔短縮・同時出現数増加・移動速度カーブの調整で対応する。敵3種目以降の追加は却下する（brief の盛らない宣言）。 |
| P-04 | 積み上がる再挑戦 | ランで得たクリスタルを Menu で恒久アップグレードに変換し、「次はもっと先へ」の意欲を作る。 | アップグレード候補を検討するとき、新アクションを付与するものではなく既存パラメータ（攻撃力・移動速度・HP）の数値強化に限定する。ラン内一時強化との二重化提案は却下する。 |

## コアループ（30秒説明）

Game シーンでヒーローがアリーナ中心に立つ。カメラは固定俯瞰（見下ろし角60〜70度、アリーナ全体を常時画角に収める）で、四方から接近する敵を同時に視認できる。自動攻撃は最寄りの1体だけを周期的に攻撃して倒すが、殲滅ペースは群れの増加ペースより意図的に低く抑えられており、敵は常に密集していく。プレイヤーは囲まれる直前に Space でダッシュ回避し、無敵窓で群れをすり抜ける。倒した敵はクリスタルを落とし、ウェーブが進むほど敵は速く・多くなる。HP が 0 になると Result へ遷移し、生存時間・撃破数・獲得クリスタルからスコアを算出。Menu でクリスタルをアップグレードに変換し、強くなって再挑戦する。

## MDA整理

- **Mechanics**: 自動攻撃（最寄り1体への単体ヒット・周期発動・照準なし・AoEではない。DPS上限は群れの増加ペースを常に下回るよう設定）／固定俯瞰カメラ（見下ろし60〜70度・アリーナ全体を常時画角内に保持）／ダッシュ回避（短い無敵窓＋クールダウン）／ウェーブスポーン（経過時間で同時数・速度が上昇）／クリスタルドロップと通貨化／Menu 恒久アップグレード購入（攻撃力・移動速度・HP）
- **Dynamics**: 敵に囲まれない位置取りの絶え間ない駆け引き（自動攻撃が殲滅しきらないため常に発生）／回避クールダウンをいつ切るかのリスク管理／ウェーブ進行に伴い生存ルートを探す圧迫感の高まり／ラン結果を踏まえた次投資先の選択（メタ判断）
- **Aesthetics**（→ P-xx）: 際どく躱せた瞬間の高揚感（→P-01）／操作不要で敵を確実に仕留め続け群れを捌いていく手応え（→P-02。brief Core Fantasy の「薙ぎ払う無双感」から単体ヒット実態に合わせて表現を調整。差分は「設計判断」参照）／包囲されていく緊張感（→P-03）／強くなって再挑戦したくなる意欲（→P-04）

## スコープ

- **In scope**: 円形アリーナ1面／ヒーロー1体（移動・ダッシュ回避・自動攻撃）／敵は基本1種（近接接近型）で成立を完結させる。2種目は行動ロジックを増やさない見た目/軽微なステータス差分の任意追加とし必須ではない（P-03: 物量と速度で難度を作る方針に合わせ、種類を増やす方向の拡張はしない）／ウェーブ難度カーブ（スポーン間隔・同時数・速度が時間で上昇）／HP・被弾・死亡判定／スコア算出（生存時間＋撃破数＋クリスタル）／ハイスコア永続記録／クリスタル通貨とラン間アップグレード3種（攻撃力・移動速度・HP）／ゲームフロー（Boot→Title→Menu→Game→Result→{Game|Menu}）／キーボード操作のみ（マウス不要）
- **Out of scope**: 武器種追加・武器進化ツリー — brief の盛らない宣言／敵3種目以降・ボス — brief の盛らない宣言／複数アリーナ・マップ探索 — セッション2〜3分に不要／FPS視点・手動照準 — P-02 と矛盾／スキン・実績の追加 — brief の盛らない宣言／ローカル/オンライン協力プレイ — brief に記載なし（brief 外の追加はしない）

## 検証したい面白さの仮説

- 敵の群れに周囲を囲まれかけた瞬間、ダッシュ回避で紙一重に切り抜けられたときにプレイヤーが思わず前のめりになる/声が出るような高揚を示すなら、このゲームは面白い。
- ラン終了後の Result 画面を見た直後、プレイヤーが操作なしに「もう一回」を選び、Menu で次に強化するパラメータを吟味してから再挑戦する行動が起きるなら、このゲームは面白い。
- 攻撃ボタンを一切押さずに自動攻撃が最寄りの敵を確実に仕留め続ける様子を見て、プレイヤーが「避けることだけに集中していても敵を切らさず捌けている」という手応え・安心感を実感し（チクチク削っているだけという退屈な感想ではなく）、単調作業ではなく無双寄りの実感を報告するなら、このゲームは面白い。

## 設計判断

<!-- テンプレート追加節（DR-CONCEPT iteration 1 revise で追加）。実装/QAの裁定に必要な決定と根拠を残す。 -->

- **攻撃形状: 単体ヒットに統一（AoE不採用）** — 自動攻撃は最寄り1体だけを攻撃する。P-01（紙一重回避）と P-03（群れ密度の圧力）は「敵に囲まれる状況が実際に発生する」ことが前提で、AoE型の全体攻撃は群れを安定的に一掃してしまい、この前提を壊すリスクが高い。単体ヒット＋DPS上限（群れの増加ペースを常に下回る）を採用し、常に一定数の敵が密集する状態を保証する。
- **カメラ: 固定俯瞰に統一** — 追従TPS視点は死角を生み、「四方から迫る群れが見えているから避けられる」という P-01 の前提を壊す。固定俯瞰（見下ろし60〜70度・アリーナ全体を常時画角内）を採用し、四方の敵を同時視認できることを担保する。
- **最大設計リスク: P-02（無双感）と P-01/P-03（囲まれる緊張）の均衡** — 自動攻撃が強すぎれば群れが一掃されP-01/P-03が成立せず、弱すぎればP-02の無双感が消える。この均衡は Checkpoint B のプロトタイプで実プレイ検証すべき最大のリスクと位置づける。GDD では自動攻撃を「間引き型」（単体ヒット・DPSに調整レンジ付きの上限を設け、ウェーブごとの敵増加ペースが常にそれを上回るよう難度カーブと連動させる）として初期値・調整レンジを規定する。
- **恒久アップグレードとセッション長の整合（GDD申し送り）** — Menu の恒久アップグレード（攻撃力・移動速度・HP）によるパワークリープが「1セッション2〜3分・エンドレス難度上昇」という想定と乖離しないよう、GDD の難度カーブはプレイヤーの恒久強化幅を吸収できる設計（例: 後半ウェーブの難度上昇率をベース強度に対する相対値で定義する）とする方針とする。具体的な数値（上昇率・上限）は GDD 側で初期値＋調整レンジとして規定する。
- **P-02 の Aesthetic 表記と brief Core Fantasy の差分** — brief の Core Fantasy は「自動攻撃で薙ぎ払う無双感」と書かれているが、iteration 1 revise で攻撃を AoE ではなく単体ヒットに確定した（P-01/P-03 の「囲まれる」前提を守るため）。この実態に合わせ MDA の Aesthetic 表記を「群れを確実に捌いていく手応え」に調整した。brief の語感（薙ぎ払う＝広範囲一掃）とは一致しないが、これは iteration 1 で明示的に下した設計判断（攻撃形状: 単体ヒットに統一）の帰結であり、brief 外の逸脱ではなく brief 内の Core Fantasy を成立させるための解釈選択である。この差分の成否（単調な作業感に落ちず無双寄りの手応えとして成立するか）を「検証したい面白さの仮説」に反証可能な形で追加した。
- **敵種数の下限と2種目の位置づけ** — P-03（物量と速度で難度を作る／敵種多様化しない）に照らし、敵は基本1種で成立を完結させることをスコープの必須ラインとする。2種目は行動ロジックを増やさない見た目・軽微なステータス差分（例: 色/シルエット差 + HPやサイズの微調整）に限定した任意追加とし、実装余力が無ければ省略してよい。
- **単体ヒット自動攻撃の当たり表現（GDD申し送り）** — 近接スイング/飛び道具/瞬間ヒットのいずれで表現するかは concept 段階では確定せず GDD で決定する。brief で確保済みの hero attack アニメクリップ・攻撃ヒット SFX と矛盾しない方式（アニメ1クリップで表現しきれる形）を選ぶこと。
