# Art Bible — Crystal Vanguard

## スタイル宣言

3D（ピクセルアートではない。brief.md「アート方向: ピクセルアート: no（3D）」と一致）のスタイライズド・トゥーンを採用する。参照様式は「モバイルソシャゲ系トゥーンレンダー（明るいファンタジー基調にネオン発光アクセント）」で、フラットセルシェード塗り＋全シルエット縁に太め（2〜3px相当）の濃紺アウトラインを施す線画スタイルとする。光源は単一のソフトなキーライト（左上前方固定）で、地面接地部・関節部にのみ柔らかいアンビエントオクルージョンの陰影を落とし、ハードな落ち影・ハッチングは使わない。ハイライトは装甲の縁とクリスタルのファセットに白色の光沢ストライプのみを許可する。

## シルエット方針

- **形での区別ルール**: hero=縦長の二足直立シルエット（肩幅より身長が明確に大きい）／敵（スウォーマー）=低く横に広い四足シルエット（体高より体長が大きく、常に地面付近に重心がある）／クリスタル=小型の幾何学的ファセット形状（尖った多面体）で地面付近に点在。3種は輪郭の縦横比だけで即座に判別できるようにする（brief.md「アート方向」節の明文指定と一致）。
- **ゲーム内表示サイズでの最小ディテール単位**: 固定俯瞰カメラ（CAMERA_HEIGHT 14m・CAMERA_PITCH 65°・ARENA_RADIUS 12m）でアリーナ全景を常時画角に収めるため、個体は画面上で小さく表示される。ディテールはシルエット輪郭と主色ブロッキングのみで判別できる粒度に抑え、顔の表情や装飾の細部など2m未満の距離でしか判別できないディテールは作り込まない（テクスチャ2048pxの解像度は近距離用の予備であり、遠景での可読性はシルエットと色ブロックが担う）。
- **前景と背景のコントラスト確保策**: 全シルエットに濃紺（`#164583`系）の太アウトラインを適用し、アリーナ床（明るい緑`#7FE850`）・void（暗い紫`#12081F`）のいずれの背景に対しても輪郭が沈まないようにする。密集時（P-03: 群れ密度の圧力でスウォーマーが最大40体同時出現）でも、敵は単一色相（紫）でまとまり hero（青）・クリスタル（シアン/マゼンタ）と混同しない配色を維持する。
- **敵とクリスタル・マゼンタの識別根拠（AR-BIBLE iteration 1 指摘1対応）**: 当初の `crystal_magenta`（#D33FD4, hue≈299.6°）は `enemy_primary`（#8B12A5, hue≈289.4°）と色相差わずか10.2°で近似しており、P-03の密集時にクリスタルが敵群に埋もれるリスクがあった。対応として `crystal_magenta` を `#E62284`（hue≈330°, val≈0.90）へ再指定し、敵主色との色相差を約40.6°に拡大した。加えて、識別は**色相のみに依存させない**設計とする: クリスタルは常にエミッシブ発光（明度val≈0.90・Bloom付きの発光ハロー）で表示され、敵（val≈0.65・非発光のマット面）とは輝度差でも即座に区別できる。さらにクリスタルは幾何学的ファセット形状で地面付近に点在する小型オブジェクトであり、四足で密集移動する敵シルエットとは形状・スケール・動き（自転+浮遊 vs 接近移動）でも重複しない。色相分離はこれら複数の識別軸の一つであり、単独の生命線ではない。

## 解像度・タイルサイズ

- コンセプト画生成解像度: 1024px（長辺。key image 候補と同一。image-to-3D の入力として使用）
- UI/2Dスプライト生成解像度: 512px（ゲーム内表示は Canvas スケールに応じて可変、縮小方式は linear。HUDアイコン・クリスタルアイコン等の2D要素用。全数アルファチャンネル必須）
- 3Dモデルテクスチャ解像度: 2048px（albedo + metallic-roughness の PBR。詳細は「3D スタイル方針」節）
- タイルサイズ: 適用なし（`tile_px: null`）— unity 3D プロジェクトのためタイルセット/atlas化は行わない（atlas化は engine=phaser のみ。assets-config.md 生成後パイプライン）。アリーナ床はタイル張りテクスチャではなくUnityプリミティブ＋単一マテリアル（グラデーション+同心円ライン）で構成する（gdd「クリスタル・アリーナ環境の視覚表現方針」参照）
- 透過方針: 全2D UIスプライトはアルファチャンネル必須（白背景PNGの出荷禁止 — assets-config.md ハード禁止事項）。3Dモデルテクスチャは不透明（アルファ不要、クリスタルの発光はUnity側のEmissiveマテリアル+Bloomで表現しテクスチャ透過には依存しない）

## アニメ方針

- **hero**: brief確保済みの3クリップのみ（ANM-01=attack / ANM-02=idle / ANM-03=run）。新規クリップは追加しない（gdd「対応資産id」節）。attackは瞬間ヒット表現のためループなし1回再生、idle/runはループ再生。被弾表現は専用クリップを持たず、マテリアルの短時間フラッシュ（ヒットフラッシュ、`#FF3B30`）で表現。死亡演出もアニメ追加なしでANM-02（idle）流用＋マテリアルフェードアウト＋簡易回転チルトのコード合成演出とする（gdd「勝敗条件」節の決定に一致）。
- **スウォーマー（通常/ヘヴィ）**: ANM-04（接近ループ）1クリップのみ。常時プレイヤー方向へ直線移動するAIのため、歩行/疾走を単一ループクリップで表現しきる（gdd「敵接近AI」節）。ヘヴィ変種は同一骨格ならANM-04を共用し、別骨格の場合もモデル差分のみでモーション自体は追加しない。
- **クリスタル**: アニメ資産を作らない。緩やかな自転（Y軸回転）と浮遊（上下バウンド）はコード側のtweenで実装する（gdd「クリスタル・アリーナ環境の視覚表現方針」節。生成MDL自体を使わないためアニメクリップも不要）。
- **HUD/フィードバック**: ウェーブ切替パルス（数値を0.3秒かけて1.0→1.3→1.0倍スケール）はUI側のtweenで実装し、アニメ資産・VFX資産は不要（gdd「HUD/フィードバック表示」節）。自動攻撃ヒット時のVFXは短命パーティクル1種（design/assets.mdで定義）を使用し、アニメクリップではなくパーティクルシステムとして実装する。

## 3D スタイル方針

3D 版の技術整合（engine=unity。1 unit = 1m、tech-stack-unity.md「資産の取り扱い」節と一致）。

- **ポリゴン予算**: hero ≤ 50,000 tri（assets-config.md 既定通り）／敵（スウォーマー・ヘヴィ変種）≤ 20,000 tri（assets-config.md の character 既定より絞る。理由: `MAX_CONCURRENT_ENEMIES=40` により画面内に最大40体が同時描画されるため hero より軽量なポリゴン予算をエンティティ単位で設定し実行時負荷を抑える）／プロップ ≤ 10,000 tri（**予約のみ・未使用**: gdd「クリスタル・アリーナ環境の視覚表現方針」決定によりクリスタルは生成モデルを使わずUnity標準プリミティブ+エミッシブマテリアルで表現するため、brief の3Dモデル上限4点中「プロップ1」枠は消費しない）／環境 ≤ 100,000 tri（**予約のみ・未使用**: アリーナ地面・境界は同上の理由でプリミティブ+マテリアルのみで構成し生成環境モデルは無い）
- **テクスチャ解像度と PBR 方針**: 2048px albedo + metallic-roughness（Meshy image-to-3D の標準PBR出力をそのまま使用）。ただし本作のスタイル方向は「フラット寄りの塗り」（brief.md アート方向）のため、albedo は低彩度差・低ディテールのフラット色面で構成された2Dコンセプト画（style_block準拠）を入力にし、metallic/roughnessは低振幅（マット寄り・光沢差を作らない）に統一してトゥーン調の質感を維持する。太いアウトライン線はコンセプト画側に描き込んだ状態でimage-to-3D生成することでベイクを試みたが、AR-ASSET iteration 1〜4（`state/reviews/assets-models-prototype.md`）の独自機械検査（HSV value分布実測: MDL-01 albedo の value<0.15画素比率0.00%）で、image-to-3Dパイプラインが2Dコンセプト画の視点依存シルエット線を3D表面テクスチャへ転写できないという既知の限界が繰り返し定量確認された（2026-07-13 Phase3 Integrate後のUnity実機screenshotでも同様に未解消と再確認）。**したがって、URP Outlineポストプロセス（インバートハル法によるバックフェース法線押し出し、またはURP法線/深度エッジ検出）の適用は「任意検討」ではなく実質必須とする**（色はダークネイビー系パレット色 `#12081F` を基準に2-3px相当幅で固定）。専用トゥーンシェーダ/ポストプロセス自体は本パイプラインの2D/3Dアセット生成ルーティング表には無いため、実装は Integration 工程（gameplay-engineer/ui-engineer 担当、CR-CODE 対象のUnityコード変更）側で行う — art-director の生成物（FBX/テクスチャ）だけでは満たせない要件であることを明記する。将来 IMG-01/IMG-02 相当のコンセプト画を再生成する機会がある場合は、プロンプトに「輪郭線はalbedoテクスチャに高コントラストの黒い縁取りとして焼き込まれ、image-to-3D変換後もUVテクスチャ上に幅2-3px相当で保持される」ことを明示し、変換後にalbedoのvalue<0.2画素比率を計測する検証ステップを追加すること（次回コンセプト画再生成時の申し送り）
- **リグ方針**: hero=humanoid（Meshy `fal-ai/meshy/rigging/multi-animation` でauto-rig。アニメクリップ語彙は idle/run/attack の3種のみ、+Z前方基準）。スウォーマー（通常/ヘヴィ）=quadruped（非ヒューマノイド。assets-config.md 3D ルーティング表「キャラクター（非ヒューマノイド/クリーチャー）」行に従い、Meshyでリグ対応困難な場合はTripo直APIへフォールバック）。アニメクリップ語彙は approach_loop（ANM-04）の1種のみ。
- **コンセプト画プロトコル**: 全モデル（MDL-01 hero / MDL-02 スウォーマー / MDL-03 ヘヴィスウォーマー任意）は、本バイブルの `style_block` を前置したキャラクター単体の2Dコンセプト画を先に生成し、それを image-to-3D の入力とする二段構え（assets-config.md「スタイル一貫性プロトコル（3D 追記）」）。hero の全ポーズ・全アニメで `character_reference`（`design/refs/crop-01-hero.png`）を共用する。
- **スケール規約**: 1 unit = 1m（Unity既定）。hero 身長 1.8m（調整レンジ 1.6–2.0m、tech-stack-unity.md「資産の取り扱い」節のヒト型基準と一致）。スウォーマーは「低く横に広い」形状方針に従い肩高 約1.0m・体長 約1.4m（体高より体長が大きい比率を維持）。up軸 +Y、前方軸 +Z（Meshy既定）。authoring-time計測値は生成時に `bbox_authoring_m` としてMANIFEST.jsonlへ記録し、Unity取込後にバウンディングボックスを再検証する（assets-config.md 生成後パイプライン 3D節）。
